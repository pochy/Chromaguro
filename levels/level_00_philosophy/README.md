# Level 0: まず検索基盤として考える

## この Level でできるようになること

Chroma を「LLM の記憶装置」ではなく、「質問に役立つ context を選ぶ検索基盤」として説明できるようになります。

この Level ではコードは書きません。最初に考え方を整えます。

## まず知るべき言葉

- **LLM**: ChatGPT のように文章を生成するモデル。
- **context**: LLM に読ませる材料。
- **検索基盤**: 質問に対して、どの文書を取り出すか決める仕組み。
- **RAG**: 検索した文書を LLM に渡して回答させる構成。

用語に迷ったら、ルートの `glossary.md` を見てください。

## なぜこれを学ぶのか

RAG では、LLM は渡された context をもとに回答します。検索で違う文書を渡すと、LLM は正しく答えられません。

```text
良い回答 = 良い文書 + 良い検索 + 良い文脈化 + 良い生成
```

Chroma はこの中の「良い検索」を支える道具です。

## 手順 1: Chroma の役割を言葉にする

次の文を読んでください。

```text
Chroma は、document を保存するだけの箱ではない。
ユーザーの質問に対して、LLM に渡すべき context を選ぶ検索基盤である。
```

最初はこれだけで十分です。

## 手順 2: 「近い」と「役に立つ」を分ける

Vector search は意味的に近い文書を探します。しかし、近い文書が質問に答えているとは限りません。

```text
質問:
この商品のアレルゲン情報は？

近いが弱い文書:
この商品は人気の焼き菓子です。保存方法は直射日光を避けてください。

答えとして有用な文書:
原材料: 小麦粉、卵、乳。アレルゲン: 小麦、卵、乳成分。
```

検索設計では、次を分けて考えます。

```text
意味的に関連しているか
質問に直接答えているか
LLM に渡すには十分に短く具体的か
参照元をユーザーに示せるか
```

## 手順 3: Search Design Canvas を見る

`examples/search_design_canvas.md` を開きます。

まだ全部埋める必要はありません。次の 4 つだけ考えてください。

```text
誰の質問に答えるのか
どの文書を検索対象にするのか
何を metadata として持つのか
検索失敗をどう分類するのか
```
